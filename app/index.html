PHP Coding Standards

Here at Debit Finance Collections Plc we aim to develop solutions and sites quickly, efficiently and effectively.

In order for us to maintain a high standard whilst still producing flexible products we need to ensure that the code we produce is of a high and consistent standard.

With this in mind, we kindly ask that you follow our coding guidelines at all times. These are based on PHP’s PSR-0, PSR-1 and PSR-2 standards but with some minor amendments.

The goal is to produce elegant and easily maintainable code.

Autoloading

All none framework based projects should implement a PSR-0 compatible autoloader. One difference being that all the class files should be located within the lib/classes directory located in the project root.

E.g. the class Example_TextHelper would be located in /lib/classes/Example/TextHelper.php.

Object Orientated Approach

To aid in the re-use and organisation of project components we opt to use classes instead of function files.

The global namespace should not define any functions and class definitions should be stored in their own file within the lib directory.

Any functions that provide general utilities, such as truncating text, can be placed into helper classes (/lib/Helper/Text.php as an example) and the function defined as a static method.

Database Access

All new code produced must use PHP’s PDO Library in favour of mysql, or mysqli. The only exception to this is when connecting to an Oracle Database where the OCI8 Functions should be used instead (at least until the PDO_OCI module becomes stable).

Cron’s

When PHP Cron’s are used, these should be setup to run in Command Line Mode, and must contain some initial checks that the script is being run in CLI Mode.

if (PHP_SAPI!=='cli') 
{
   die("Cron must be run in CLI Mode");
}

The Cronlock Library (developed by Andrew Radford) should also be used, which will allow for logging or cron’s for debugging, along with stopping cron’s from overlapping. Code is available on GitHub

Naming Conventions

All class names are Capitalised.

Underscores in class names indicate directory separators and function names are camelCase.

Private variables within classes should be prefixed with an underscore.

Class private variables are to be prefixed with an underscore and must omit the var keyword.

All PHP keywords – such as true, false, null – are to be lowercase.

File Responsibilities

A file can either load and output or define, not both. This is to say that any file that defines a class should not output or change anything globally. This allows the file to be used much more freely without worrying about dependencies. When writing a class, this should be able to be added to any project without any alteration, and therefore should be self-sufficient.

Global Variables & the Global Keyword

PHP makes it very easy to programme quickly but the downside of this is that it can quickly allow people to produce un-mangeable code. A strong cause of this are global variables and the global keyword.

The use of global variables is discouraged and even more so the use of the global keyword to reference global parameters. There is always a better option that may require slightly more work but the up-side is the speed at which it can be extended, tested and refactored later on.

The easy option is rarely the best option.

File Formatting & Encoding

All PHP files should be encoded using UTF-8 without BOM. All PHP files must have a single .php extension. The only time other extensions are acceptable is if following a clearly defined pattern (often found in frameworks for template files).

When defining classes or functions the curly brackets must be on a separate line. Similarly this is the same as conditional statements and loops.

class MyClassName
{
	public function __construct()
	{
	    if (true)
		{
	        // true
	    }
		else
		{
	        // false
	    }
	}
}

All files must be correctly indented using tabs and any files that contain only PHP should leave out the php closing tag. This prevents any whitespace at the end of the file.
